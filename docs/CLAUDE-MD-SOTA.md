# CLAUDE.md Lifecycle Guidelines

Authoritative reference for `/ignite` target project CLAUDE.md generation. **Seed version**: web-sourced best practices only. Run `/refresh-guidelines` to produce `CLAUDE-MD-SOTA.enriched.md` (seed + `/insights` data, gitignored).

## Purpose & Scope

This **seed file** defines the standards for CLAUDE.md files that `/ignite` produces for target projects. It contains web-sourced best practices only:

1. **Web-sourced best practices** — official Anthropic docs (Tier 1), established community guides (Tier 2), and community templates (Tier 3).

When `/refresh-guidelines` runs, it produces `docs/CLAUDE-MD-SOTA.enriched.md` (gitignored) which merges this seed with:

2. **`/insights` tips** — real behavioral patterns discovered from actual Claude Code sessions, accumulated by the user.

`/ignite` reads the enriched version if present, falling back to this seed.

**How `/ignite` consumes this document:**

| Part | /ignite Usage |
|------|---------------|
| Part 1: Structural Standards | Determines CLAUDE.md skeleton — sections, order, size budget |
| Part 2: Content Guidelines | Decides what goes in each section and what to exclude |
| Part 3: Integration Guidelines | Decides what goes in CLAUDE.md vs hooks vs skills vs rules |
| Part 4: Behavioral Patterns | Selected and adapted based on project characteristics |
| Part 5: Maintenance Guidelines | Brief maintenance section or @import reference |

## Part 1: Structural Standards

### 1.1 File Hierarchy

CLAUDE.md operates within a multi-level memory hierarchy. `/ignite` should generate the project-level file and advise on the others:

- **`CLAUDE.md`** (project root) — Main project instructions, read every session. Always generated by `/ignite`.
- **`CLAUDE.local.md`** (project root, gitignored) — Personal preferences and local overrides. Not generated; mentioned in maintenance section.
- **`~/.claude/CLAUDE.md`** (user home) — Global instructions across all projects. Not generated; out of scope.
- **`.claude/rules/`** — Modular topic-specific rule files with optional glob scoping (e.g., rules that apply only to `*.ts` files). Used for stable, categorical rules.
- **Child-directory `CLAUDE.md`** — Scoped instructions for subtrees (e.g., `src/api/CLAUDE.md`). Used for large monorepos.
- **Organization-managed `CLAUDE.md`** — Centrally deployed via MDM, Ansible, or Group Policy for consistent team-wide rules across all developer machines.

*Sources: T1-001, T1-003, T3-002*

### 1.2 @-Imports for Modularity

Use `@path/to/file.md` to import shared instructions rather than duplicating content. Keeps the root CLAUDE.md concise and avoids drift between copies. Examples: `@docs/git-instructions.md`, `@.claude/testing-conventions.md`.

*Source: T1-001, T1-002*

### 1.3 Recommended Sections

Generated CLAUDE.md files should include these sections in order:

1. **Project Overview** — What the project is, in 1-3 sentences.
2. **Tech Stack** — Languages, frameworks, databases, infrastructure.
3. **Common Commands** — Build, test, lint, deploy commands (non-obvious ones only).
4. **Project Structure** — Key directories and their purposes.
5. **Coding Conventions** — Style rules, naming, patterns specific to this project.
6. **Workflow Preferences** — How to approach changes, verification steps.
7. **Installed Entities** — (added by `/ignite`) Agents, skills, hooks installed.
8. **Known Gaps** — (added by `/ignite`) Technologies not covered by catalog.

*Sources: T1-003, T2-003, T2-004*

### 1.4 Formatting Rules

- **Bullets over prose** — Scannable lists are followed more reliably than paragraphs.
- **Specific over vague** — `pytest tests/ -v` beats "run the tests".
- **Headings for scanning** — Claude parses structured documents more effectively.
- **Keep concise** — Shorter files are more reliably followed. General consensus: <300 lines max; HumanLayer keeps theirs under 60 lines. Every line should earn its place.
- **Repeat critical rules** — Emphasis (bold) or repetition at multiple points improves compliance with the most important instructions.
- **Emphasis keywords** — Use `IMPORTANT` or `YOU MUST` to improve adherence on critical rules. Standard phrasing is easier for the model to deprioritize.

*Sources: T1-002, T1-003, T2-001, T3-002*

### 1.5 Placement Decision Guide

| Scope | File | Examples |
|-------|------|----------|
| **Global** | `~/.claude/CLAUDE.md` | "Always run tests", "ask before committing" |
| **Project** | `CLAUDE.md` | Tech stack, structure, commands, conventions |
| **Local** | `CLAUDE.local.md` (gitignored) | Terminal prefs, personal MCP servers |

Override order: local > project > global. Don't duplicate rules across levels.

*Source: T3-002*

## Part 2: Content Guidelines

### 2.1 What to Include

Structure content around three questions — **WHAT** (tech, stack, project structure), **WHY** (project purpose, component roles), **HOW** (build/test/deploy commands, verification steps).

- Non-obvious build, test, and lint commands.
- Architectural decisions and their rationale.
- Naming conventions specific to this project.
- Gotchas and known issues (e.g., "migrations must be run before tests").
- Environment quirks (e.g., "requires Node 20+, not 22").
- Domain-specific terminology and vocabulary.
- Type hint/annotation policies.
- Test framework details and fixture locations.
- Import ordering and style preferences.
- Error handling patterns.
- Custom CLI commands, monorepo navigation, non-standard file locations.
- Team workflows — step-by-step procedures for common tasks (e.g., creating API endpoints: plan → confirm → implement → test → verify). Keeps agent output consistent with team standards.
- **Prefer pointers to copies**: Don't embed code snippets in CLAUDE.md or context files — they go stale. Use `file:line` references to point Claude at the authoritative source code.

*Sources: T1-002, T1-003, T2-001, T2-003, T2-004*

### 2.2 What to Exclude

- Standard language conventions Claude already knows (e.g., "use const in JS").
- Linter/formatter configurations — enforce via hooks, not CLAUDE.md.
- Stale code snippets that will drift from the actual codebase.
- Task-specific instructions (use skills or conversation context instead).
- Secrets, credentials, or environment variables with real values.
- Generic advice ("write clean code", "follow best practices").

*Sources: T1-002, T2-001*

### 2.3 Anti-Patterns

- **Kitchen sink**: Dumping everything into CLAUDE.md. Research shows LLMs reliably follow ~150-200 instructions; Claude Code's system prompt already uses ~50, leaving limited headroom. More instructions = uniform quality decay across **all** of them — not just newer ones, all instructions degrade equally (smaller models decay exponentially, frontier thinking models linearly). Extract stable rules to `.claude/rules/` or hooks.
- **Vague instructions**: "Write good code" is useless. "Use pytest fixtures from `tests/conftest.py`, run `make test` before committing" is actionable.
- **Over-specification**: Documenting what Claude already knows adds noise without value.
- **No curation**: Auto-generating CLAUDE.md without human review leads to bloated, generic files.
- **Non-universal instructions**: Claude Code wraps CLAUDE.md in a `<system-reminder>` stating "this context may or may not be relevant." If content isn't universally applicable to the current task, Claude may ignore it entirely. Keep all instructions focused and broadly relevant.
- **Personality prompts**: "Be a senior engineer" or "Think step by step" wastes tokens. Claude Code already has strong system-level instructions — role-playing directives add noise.
- **Blanket @-imports**: Writing `@docs/api-guide.md` embeds the entire file into context every session. Instead, *describe when* to read it: "For Stripe integration issues, see `docs/stripe-guide.md`." Let Claude decide when context is needed.

*Sources: T2-001, T2-003, T3-002*

## Part 3: Integration Guidelines

### 3.1 CLAUDE.md vs Hooks

- **CLAUDE.md** = advisory behavioral guidance. Claude follows it as best-effort context.
- **Hooks** = mandatory automated enforcement. Run deterministically on events (pre-commit, file save).
- **Rule of thumb**: If a rule can be checked mechanically (formatting, linting, type checking), use a hook. If it requires judgment (architecture, naming, approach), use CLAUDE.md.

*Source: T1-001*

### 3.2 CLAUDE.md vs Skills

- **CLAUDE.md** = always-active background context read every session.
- **Skills** = on-demand task workflows invoked explicitly (e.g., `/tdd`, `/code-review`).
- **Rule of thumb**: Don't put procedural multi-step workflows in CLAUDE.md. Use skills for task-specific procedures, CLAUDE.md for universal project guidance.

*Source: T1-002*

### 3.3 CLAUDE.md vs .claude/rules/

- **CLAUDE.md** = universal project guidance, always loaded.
- **`.claude/rules/`** = modular, topic-specific rule files. Support glob patterns for file-type scoping (e.g., `*.{ts,tsx}` via brace expansion). Rules organized in subdirectories (`frontend/`, `backend/`), discovered recursively. Symlinks supported for sharing rules across projects. User-level rules in `~/.claude/rules/` apply globally. Best practices: keep each file focused on one topic, use descriptive filenames, use conditional `paths` sparingly, organize related rules with subdirectories.
- **Rule of thumb**: Stable, categorical rules (language-specific conventions, security policies) go in rules/. Project-wide context (overview, structure, commands) stays in CLAUDE.md.

*Sources: T1-001, T1-002*

### 3.4 MCP Server Integration

Document configured MCP servers and instruct Claude when/how to use them. MCP tools are configured via JSON; CLAUDE.md tells Claude the intent and workflow. Example: "Use the Shadcn MCP to browse component registries before creating UI elements." Remind agents to check MCP tools before falling back to manual approaches.

*Sources: T2-003, T1-002*

### 3.5 Headless & Batch Mode

Claude Code can run non-interactively (`claude -p "task"`) for batch processing. Use `--allowedTools` to scope permissions per invocation. Relevant for projects with repetitive multi-file tasks (migrations, bulk formatting, parallel validation).

*Source: T1-002*

### 3.6 @-Import Strategy

Reference shared instruction files to keep CLAUDE.md lean and modular:
- `@docs/git-instructions.md` — Git workflow rules.
- `@.claude/testing-conventions.md` — Test patterns.
- Personal overrides: `@~/.claude/my-project-instructions.md`.
- **Context file approval**: List context files with brief descriptions and instruct Claude to present which ones it wants to read for approval before loading. Gives explicit control over what context enters each session.

*Sources: T1-001, T1-002, T2-001, T2-004*

### 3.7 Custom Subagents

Define specialized assistants in `.claude/agents/` with their own tool set, model, and system prompt. Subagents run in isolated context for focused tasks (security review, code analysis, documentation). CLAUDE.md should document available subagents and when to invoke them. Example: "Use a subagent to review this code for security issues."

*Source: T1-002*

### 3.8 Plugins

Use `/plugin` to browse and install community-provided plugins that bundle skills, hooks, subagents, and MCP servers. Eliminates manual configuration for common setups. CLAUDE.md can document which plugins are active and any project-specific usage notes.

*Source: T1-002*

## Part 4: Behavioral Patterns

### 4.1 Verification Criteria

Give Claude specific verification commands rather than vague "make sure it works":
- `npm test`, `cargo clippy`, `mypy .`, `pytest tests/ -v`
- Include the exact commands in the Common Commands section of generated CLAUDE.md files.

*Source: T1-002*

### 4.2 Explore-Then-Plan Workflow

Instruct Claude to read existing code before modifying it. Prevents blind refactors that break established conventions. Document environment constraints upfront (existing entrypoints, file structures, mode limitations) to avoid false-start iterations. Example instruction: "Always read the existing implementation before proposing changes."

*Source: T1-002*

### 4.3 Code Style Preferences

Document specific style rules Claude should follow — import ordering, destructuring preferences, naming conventions. These are project-specific and not inferable from the language alone.

*Source: T1-002*

### 4.4 Scope Discipline

"Do what has been asked; nothing more, nothing less." Prevents scope creep in AI-assisted coding. Generated CLAUDE.md files should include a scope discipline rule when the project values minimal, focused changes.

*Source: T3-001*

### 4.5 File Discipline

"NEVER create files unless explicitly needed." Prevents unnecessary file proliferation. Relevant for projects that want tight control over repository structure.

*Source: T3-001*

### 4.6 Parallel Sessions

Use multiple Claude Code sessions for quality workflows. Fan out across files with `claude -p` for batch operations; use `--allowedTools` to scope permissions per invocation.

*Source: T1-002*

### 4.7 Writer/Reviewer Pattern

Dedicate separate sessions to distinct roles: a Writer session implements, a Reviewer session reviews with fresh context (avoids bias toward own code). Also applies to TDD: one session writes tests, another writes code to pass them.

*Source: T1-002*

## Part 5: Maintenance Guidelines

### 5.1 Review Triggers

Update CLAUDE.md when:
- Recurring mistakes appear across sessions (abstract the fix into a rule).
- Tech stack changes (new framework, new build tool, new test runner).
- Onboarding new team members (if they need to explain something, add it).

*Sources: T1-003, T2-001*

### 5.2 Self-Improving Pattern

1. **Observe** a mistake → 2. **Abstract** into a rule → 3. **Add** to CLAUDE.md or `.claude/rules/` → 4. **Verify** next session.

Shortcut: after every correction, end with "Update CLAUDE.md so you don't make that mistake again." Use `/memory` to open any memory file in your system editor for extensive edits.

*Sources: T1-001, T1-003, T3-002*

### 5.3 Version Control

- Track CLAUDE.md in git like any other code file.
- Review changes in pull requests — team members should contribute rules from their sessions.
- Use `CLAUDE.local.md` (gitignored) for personal preferences that shouldn't affect the team.

*Source: T2-001*

### 5.4 Bootstrapping

Use `claude /init` for a baseline, but always curate every line. Start simple, expand based on friction points. `/ignite` replaces this with catalog-driven generation and human review.

*Sources: T1-002, T1-003, T2-001*
